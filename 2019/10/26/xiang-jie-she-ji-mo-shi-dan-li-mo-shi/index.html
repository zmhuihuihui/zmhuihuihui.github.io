详解设计模式—单例模式
单例模式
设计模式
发布日期:   2019-10-26

单例模式概述

单例模式是23种设计模式中较为简单的一种。其模式保证实体类只能创建并返回唯一的一个对象，其结构需要保证 构造器私有化 和 静态方法返回对象，通过该类的唯一访问方式获得其唯一的对象，可以直接访问此方法，不需要实例化该类的对象。

实现方式

  • 方法一：饿汉模式
public class Singleton {

    //构造器私有化
    private Singleton(){}

    //该类的唯一对象
    private final static Singleton singleton = new Singleton();

    //返回此对象
    public static Singleton getInstance(){
        return singleton;
    }
}

这种创建单例模式的方式体现在一个“饿”字，意思就是提前生成好该类的唯一对象，当用户想要此对象时，通过静态方法直接返回该对象。
优点：线程安全
缺点：如果用户用不到此类，则该创建对象的内存空间就被浪费了

  • 方法二：懒汉模式
public class Singleton {

    //构造器私有化
    private Singleton(){}

    //该类的唯一对象
    private static Singleton singleton ;

    //返回此对象
    public static Singleton getInstance(){
        if(singleton == null) return new Singleton();
        else return singleton;
    }
}

和饿汉模式类似，但其不同点在于突出“懒”字，不事先创建好该类的对象，而是当用户需要时再创建此对象。
优点：节省内存空间
缺点：线程不安全（可以通过synchronized进行加锁来解决，但会阻塞其他线程影响效率）

  • 方法三：双重锁模式
public class Singleton {

    //构造器私有化
    private Singleton() {
    }

    //该类的唯一对象
    private volatile static Singleton singleton;

    //返回此对象
    public static Singleton getInstance() {
        if (singleton == null) {
            synchronized (Singleton.class) {
                if(singleton == null) singleton = new Singleton();
            }
        }
        return singleton;
    }
}

为了保证线程安全可以使用双重锁机制。可以通过对创建对象的过程加锁实现线程安全，在上述实现中，没有对getInstance()方法直接加锁是因为这个方法是静态方法，对静态方法加锁相当于对整个类加锁。因此在if判断后加锁，如果多个线程进入if中，只会有一个线程来创建对象，其他对象会阻塞。

但仅仅通过加锁会出现一个问题，无法解决可能出现的重排序问题。重排序 是指编译器和处理器为了优化程序性能而对指令序列进行重新排序的一种手段。例如上述在创建Singleton对象时，正常创建流程分为三步：
1、为Singleton对象分配内存
2、Singleton对象执行初始化
3、让singleton引用指向Singleton对象

但有一定几率编译器可能会对上述步骤进行重排序，会先执行第三步再执行第二步。这时就会出现问题，如果一个线程在创建对象的过程中执行了第三步而还没执行第二步，此时第二个线程在if判断时就为非空，直接就返回了此对象，然而事实这个对象还没被初始化，因此就会抛出异常。

为了避免上述产生的问题，需要在Singleton对象前添加volatile关键词，有了这个关键词就可以禁止编译器在创建该类时使用重排序。

优点：线程安全，节省内存空间

  • 方法四：静态内部类
public class Singleton {

    //构造器私有化
    private Singleton() {
    }

    //静态内部类创建对象
    private static class SingletonInner{
        private static final Singleton singleton = new Singleton();
    }

    //返回此对象
    public static Singleton getInstance() {
        return SingletonInner.singleton;
    }
}

静态内部类只会在getInstance()方法被调用时才会初始化，解决了内存浪费的情况，同好保证了多线程的安全访问。
优点：线程安全，节省内存空间

  • 方法五：枚举类
public enum EnumSingleton {
    INSTANCE;

    public static EnumSingleton getInstance(){
        return INSTANCE;
    }
}

枚举类自身优势可以保线程安全，同时隐藏了私有构造器，其中只要定义一个对象就能实现单例模式。实现较为简单。在单例模式的所有实现中，推荐使用此方法来实现。

单例模式中存在问题

在实际业务场景中，使用单例模式的过程中可能会有以下几个问题。

1、使用序列化和反序列化会破坏单例模式。

序列化即将对象转换为字节序列。反序列化即将字节序列转换为对象，使用序列化和反序列化可以使对象用于进程之间的通讯传输。在这个过程中，单例模式会出现问题(枚举类方法除外)，接下来通过饿汉模式来模拟下这个出错的过程，其代码如下：

//饿汉模式，实现序列化接口
public class HungrySingleton implements Serializable {

    //构造器私有化
    private HungrySingleton() {
    }

    //该类的唯一对象
    private static HungrySingleton singleton = new HungrySingleton();

    //返回此对象
    public static HungrySingleton getInstance() {
        return singleton;
    }
}
public class Main {
    public static void main(String[] args) throws Exception {

        HungrySingleton oldSingleton = HungrySingleton.getInstance();

        //序列化，将该对象存入文件
        ObjectOutputStream oos = new ObjectOutputStream(new FileOutputStream("test.txt"));
        oos.writeObject(oldSingleton);
        oos.close();

        //反序列化，从文件中取出该对象
        ObjectInputStream ois = new ObjectInputStream(new FileInputStream("test.txt"));
        HungrySingleton newSingleton = (HungrySingleton) ois.readObject();
        ois.close();

        System.out.println(oldSingleton);
        System.out.println(newSingleton);
    }
}

在主函数中，通过ObjectOutputStream流将对象存入test.txt文件，然后通过ObjectInputStream流再将对象取出来。按照我们预期的结果，应该oldSingleton和newSingleton为同一对象，但实际却是两个不同对象：

那么为什么与我们预期的结果不一样呢？因为序列化不是本篇文章的重点，所以这里就简单分析一下。

在上述代码中通过ObjectInputStream的readObject()方法来获取文件中的对象，而实际在readObject()方法源码中，会判断该Class是否实现了serializable或externalizable接口，如果实现了则会通过反射机制重新创建一个新的实例并返回。所以就会出现我们从文件中拿到的对象并不是之前放入的对象，那么该怎么防止序列化对单例模式的破坏呢？

只需要在类中添加一个readResolve()方法，其实现如下：

//饿汉模式，实现序列化接口
public class HungrySingleton implements Serializable {

    //构造器私有化
    private HungrySingleton() {
    }

    //该类的唯一对象
    private static HungrySingleton singleton = new HungrySingleton();

    //返回此对象
    public static HungrySingleton getInstance() {
        return singleton;
    }

    //防止序列化的破坏
    public Object readResolve(){
        return singleton;
    }
}


仅通过在类中添加readResolve()方法，就可以达到防止序列化对单例模式的破坏。其原理如下，在上面讲到readResolve()方法会通过反射拿到一个新的对象，但这个对象不会立马返回，之后还会判断该类中是否有readResolve()方法（方法名不可改），如果有就会调用该方法然后拿到我们创建好的唯一对象并返回。这样就确保了在序列化过程中单例模式的正常使用。以下截取一段ObjectInputStream类中readOrdinaryObject()方法中的关键源码,加深下理解：

private Object readOrdinaryObject(boolean unshared)throws IOException
{
     ……

  Object obj;
  try {
      obj = desc.isInstantiable() ? desc.newInstance() : null;  //isInstantiable返回true，通过反射拿到新对象
  } catch (Exception ex) {
      throw (IOException) new InvalidClassException(desc.forClass().getName(),"unable to create instance").initCause(ex);
  }

     ……

  if (obj != null && handles.lookupException(passHandle) == null
      && desc.hasReadResolveMethod()){   //通过这个名字就可以看出判断是否有readResolve()方法
        Object rep = desc.invokeReadResolve(obj);   //调用我们写的readResolve()拿到唯一对象

            ……

        handles.setObject(passHandle, obj = rep);  //将上面obj重新指向这个唯一对象
  }

  return obj;
}

2、使用反射会破坏单例模式。

反射可以通过Class对象直接创建实例（枚举类方法除外），这违背了单例模式创建唯一对象的原则。因此，需要在单例模式中添加反射防御，在懒汉模式和匿名内部类两个方式中，可以有效的防止反射。但在饿汉模式和双重锁模式中无法有效的防御。原因是前两个方法中的对象在类加载的时候就分配好，而后两种则得等到使用时才分配内存。

以饿汉模式为例，如何防止反射：

public class HungrySingleton implements Serializable {

    //构造器私有化,并且禁止反射
    private HungrySingleton() {
        if(singleton != null){
            throw new RuntimeException("禁止反射");
        }
    }

    //该类的唯一对象
    private static HungrySingleton singleton = new HungrySingleton();

    //返回此对象
    public static HungrySingleton getInstance() {
        return singleton;
    }

    //破话序列化
    public Object readResolve(){
        return singleton;
    }
}

在私有构造器中，如果反射想要通过构造器来生成一个新的对象则会抛出异常。在主函数中测试如下：

public class Main {
    public static void main(String[] args) throws Exception {
        //获得Class对象
        Class clazz = HungrySingleton.class;
        //获得构造器对象
        Constructor constructor = clazz.getDeclaredConstructor();
        //开放构造器权限
        constructor.setAccessible(true);

        HungrySingleton singleton1 = (HungrySingleton) constructor.newInstance();
        HungrySingleton singleton2 = HungrySingleton.getInstance();
        System.out.println(singleton1);
        System.out.println(singleton2);

    }
}


同样，可以在静态内部类中添加反射防御。而在饿汉模式和双重锁模式却无法防御，因为对象内存的分配时间是不确定的，无法确定是反射先创建对象还是方法先创建对象。

综上所述，在使用单例模式过程中应该时刻注意序列化和反射攻击，在《Efftctive Java》这本书中，推荐使用枚举类来实现单例模式，因为枚举类具有天生防御序列化和反射的能力，并且语法简单，容易使用。
