详解设计模式—简单工厂
工厂模式
设计模式
发布日期:   2019-10-26
文章字数:   973
阅读时长:   3 分

简单工厂严格来说并不属于GoF的23种设计模式之一，但工厂方法和抽象工厂都是在简单工厂的基础上衍生出来。所以，把简单工厂放入设计模式之中来讲解是十分有必要的。

情景介绍

假设现在市面上有两种型号的电脑，其型号分别为Windows和Mac。现在客户想购买电脑，应该如何设计这个场景呢？

首先，为了考虑到以后的可扩展，我们可以抽象出一个电脑类，不同型号的电脑分别继承这个抽象类。其代码如下：

//抽象电脑类
public abstract class Computer {
    //显示电脑型号
    public abstract void displayModel();
}
public class winComputer extends Computer {

    @Override
    public void displayModel() {
        System.out.println("This is windows");
    }
}
public class macComputer extends Computer {

    @Override
    public void displayModel() {
        System.out.println("This is mac");
    }
}

然后我们可以通过 多态 来得到我们想要的电脑，其代码如下：

public class Main {
    public static void main(String[] args) {
        Computer computer = new macComputer();
        computer.displayModel();  // mac
    }
}

想要不同型号的电脑，客户端（Main类）只需实现不同型号的电脑子类就可以。但这样会存在一个很大的问题：耦合过大。来看其UML图：

可以从上图中看到，macComputer和winComputer分别继承Computer，Main类（客户端）依赖Computer，同时也需要依赖所有的Computer子类，这样就会导致客户端与实现类之间的耦合过大。迪米特法则 告诉我们一个类应该对其他的类保持尽可能少的了解，类与类之间不应该有过大的耦合关系。因此，在这种情况下就可以使用简单工厂来解耦。

简单工厂

那么简单工厂如何实现解耦？首先来看下使用简单工厂后的UML图：

从图中看到，多了一个computerFactory类，顾名思义这个类就是专门生产各种型号电脑的工厂。Main类（客户端）只需与这个工厂打交道，只依赖于这个工厂和Computer抽象类，并不关心具体有哪些型号的电脑，也不需要依赖所有型号的电脑，这就是简单工厂。

其定义如下：

简单工厂：由一个工厂对象决定创建哪一种产品类的实例

简单工厂有以下几个特点：

  • 客户端只需要给工厂传入参数，并不关心如何创建对象
  • 工厂类肩负创建并返回对象的职责

重构场景

然后我们来使用简单工厂来重构场景。Computer抽象类和Computer实现子类保持不变，需要添加computerFactory类。这里分别通过两种方式来实现computerFactory类。

  • 方法一：String参数判断
public class computerFactory {

    //返回computer实例
    public static Computer getComputer(String type){

        if("mac".equalsIgnoreCase(type)){
            return new macComputer();
        }else if("windows".equalsIgnoreCase(type)){
            return new winComputer();
        }
        return null;
    }
}
public class Main {
    public static void main(String[] args) {
        Computer computer = computerFactory.getComputer("mac");
        computer.displayModel();   // mac
    }
}

通过Main（客户端）传入的String参数，来判断该返回什么类型的电脑。使用这一种方法虽然可以实现简单工厂，但可扩展性低，当有更多型号的电脑加入时，需要添加修改原来代码中的if判断，违反了 开闭原则 。因此，第二种方法进行了改进。

  • 方法二：反射
public class computerFactory {

    //返回computer实例
    public static Computer getComputer(Class clazz) {
        Computer computer = null;
        try {
            computer = (Computer) Class.forName(clazz.getName()).newInstance();
        } catch (InstantiationException e) {
            e.printStackTrace();
        } catch (IllegalAccessException e) {
            e.printStackTrace();
        } catch (ClassNotFoundException e) {
            e.printStackTrace();
        }
        return computer;
    }
}
public class Main {
    public static void main(String[] args) {
        Computer computer = computerFactory.getComputer(macComputer.class);
        computer.displayModel();  // mac
    }
}

不同于第一种方法的传入String参数，而是传入Class对象，通过反射机制可以动态的生成具体的Computer实现类，这样当有新的型号电脑出现时，不需要修改工厂里的方法，大大的提高了可扩展性。
