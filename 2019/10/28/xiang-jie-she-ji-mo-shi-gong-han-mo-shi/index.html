详解设计模式—工厂模式
工厂模式
设计模式
发布日期:   2019-10-28
文章字数:   1.6k
阅读时长:   5 分

工厂模式概述

工厂模式分为两种：工厂方法和抽象工厂。由于简单工厂并不属于GoF的23种设计模式之一，所以我没有把简单工厂列入工厂模式。但不管是工厂方法还是抽象工厂都是在简单工厂的基础上增加了可扩展性。他们都属于设计模式中创建型的模式，其主要功能是将对象的实例化过程抽取出来，目的就是为了降低系统中代码的耦合度，增强系统的可扩展性。

情景回顾

在简单工厂中，我们模拟了用户购买电脑的场景，用户直接与ComputerFactory类打交道就可以拿到自己想要型号的电脑，其UML图如下：

但在传统的简单工厂的实现中，如果想要添加不同型号的电脑就需要在ComputerFactory类中修改，这违背了 开闭原则 ，不利于系统的可扩展性。因此，针对不同的 产品等级 的可扩展，提出了工厂方法。

工厂方法

工厂方法的核心思想就是：

在简单工厂的基础上，让类的实例化推迟到子类实现。

对比简单工厂可以发现，在简单工厂中对象的实例化操作是在ComputerFactory类中进行的。而工厂方法的思想就是这个实例化过程推迟到了ComputerFactory子类实现，其UML图如下：

可以看到，工厂方法将ComputerFactory类提升到了抽象类，其子类为不同电脑商家的工厂，这样保证了各自的工厂产各自的电脑，而不像简单工厂中的ComputerFactory类可以产出所有商家的电脑，当有新类型的电脑出现，只要添加生产该类型电脑的工厂即可，不需要对原来的代码进行修改。因此使用工厂方法，满足了开闭原则，大大提高了可扩展性。
工厂的具体的实现如下：

//生产电脑的抽象生厂
public abstract class computerFactory {

    public abstract Computer createComputer();
}
//生产mac的工厂
public class macFactory extends computerFactory {
    @Override
    public Computer createComputer() {
        return new macComputer();
    }
}
//生产windows的工厂
public class winFactory extends computerFactory {
    @Override
    public Computer createComputer() {
        return new winComputer();
    }
}

客户端实现如下：

public class Main {
    public static void main(String[] args) {
        computerFactory factory = new macFactory();  //生产何种电脑取决于实现何种工厂
        Computer computer = factory.createComputer();
        computer.displayModel();  //mac
    }
}

工厂方法有以下几个的优点：

  • 用户只需要关心具体的工厂，而不用关心产品创建的细节
  • 符合开闭原则的设计，提高可扩展性

同样，工厂方法有以下几个的缺点：

  • 工厂类的个数容易过多，增加复杂度
  • 增加了系统的抽象性与理解难度

情景升级

现在在原有情景的基础上，用户又想购买手机，市面上同样有两种类型的手机分别为Apple和Android。

之前的工厂方法，我们知道应该要创建一个phoneFactory抽象类，然后实现不同的手机工厂生产不同的手机，流程同之前如出一辙。但如果此时用户想要购买苹果商家的pad、耳机、鼠标、键盘……一系列的东西，如果还使用工厂方法就会产生“类爆炸”的情况，我们需要创建大量的工厂类，这大大的提高了代码的复杂度和冗余度。因此，面对的不同 产品族 的可扩展性，提出了抽象工厂。

抽象工厂

抽象工厂的定义如下：

抽象工厂提供了一个创造一系列相关对象或相互依赖对象的接口

何为同一系列的相关对象？在上面的场景中，可以理解为同一家公司的产品。也就是说如果是同一家公司的产品，就可以封装在一起，而没有必要像工厂方法一样为每一个产品创建一个工厂。其UML图如下：

从UML中看出，现在多了一个产品手机。不同与工厂方法的是，并没有创建computerFactory和phoneFactory类。而是一个ElectronicFactory抽象类，其子类为不同的公司的工厂，不同公司负责生产各自公司的产品。这样，如果想要某个公司的产品，直接实现该对应公司的工厂就可以。

工厂的具体实现如下：

//抽象工厂
public abstract class ElectronicFactory {
    //生产电脑
    public abstract Computer createComputer();
    //生产手机
    public abstract Phone createPhone();
}
//苹果公司的工厂
public class AppleFactory extends ElectronicFactory {

    @Override
    public Computer createComputer() {
        return new MacComputer();
    }

    @Override
    public Phone createPhone() {
        return new ApplePhone();
    }
}
//微软公司的工厂
public class MicrosoftFactory extends ElectronicFactory {

    @Override
    public Computer createComputer() {
        return new WinComputer();
    }

    @Override
    public Phone createPhone() {
        return new AndroidPhone();
    }
}

客户端的实现如下：

public class Main {
    public static void main(String[] args) {
        ElectronicFactory factory = new AppleFactory();
        Computer computer = factory.createComputer();
        Phone phone = factory.createPhone();

        computer.displayModel(); //mac
        phone.displayModel();  //apple phone

    }
}

抽象工厂有以下几个优点：

  • 用户只需要关心具体的工厂，而不用关心产品创建的细节
  • 将一个系列的产品族统一到一起创建

同样，抽象工厂有以下几个的缺点：

  • 想要在一个产品族中扩展新的产品需要修改原来工厂中的代码，不符合开闭原则。
  • 增加了系统的抽象性与理解难度

对比工厂方法和抽象工厂

此时，我们来对比下工厂方法和抽象工厂。通过上述的例子我们发现，其实工厂方法和抽象工厂的关注点是不一样。

工厂方法关注的是 产品等级，何为产品等级？即不同商家产出的同类型的商品属于统一产品等级。
抽象工厂关注的是 产品族，何为产品族？即同一商家产出的不同类型的商品属于同一产品族。

如果确定了产品族和其产品等级，那么也就唯一确定了产品。

因此，对于不同的业务场景，工厂方法和抽象工厂各有优劣。例如，如果业务场景更多的强调的是不同的商品，那么使用工厂方法会更好一些；如果业务场景更多的强调的是不同的公司下的产品，则抽象工厂更好一些。
