装饰者模式概述

装饰者模式是Gof的23种设计模式之一，在日常的使用中也比较常见。那么何为装饰者模式？通过“装饰”一词大概可以猜出，就是对原有的对象进行装饰。

装饰者模式概念如下：

在不改变原有对象的基础上，将功能附加在对象上

先对装饰者模式有个大概的概念，接下来通过场景进一步理解。

场景引入

这次模仿店家买蛋糕的一个场景，首先店家有一个基础的蛋糕需要销售，其蛋糕实现如下：

public class Cake {

    //蛋糕的配方
    public String getDesc(){
        return "蛋糕";
    }

    //蛋糕的价格
    public int price(){  
        return 100;
    }
}

顾客想买蛋糕时，通过这个蛋糕的getDesc()方法可以知道这个蛋糕有哪些东西，以及通过price()方法可以知道它的价格。而单单只有蛋糕吸引不了顾客，商家应该在蛋糕的基础上添加一些别的东西。如果不使用装饰着模式，一般我们会想到用继承来实现：

public class CakeWithCream extends Cake {

    //在原先蛋糕上加一些奶油
    @Override
    public String getDesc() {
        return super.getDesc()+" 加一点奶油";
    }

    //在原先价格上加20
    @Override
    public int price() {
        return super.price()+20;
    }
}

通过继承可以在蛋糕的基础上加一些奶油，同样也可以在加一些奶油的基础上加一些水果：

public class CakeWithCreamFruit extends CakeWithCream{

    //在原先蛋糕上加一些水果
    @Override
    public String getDesc() {
        return super.getDesc()+" 加一点水果";
    }

    //在原先价格上加30
    @Override
    public int price() {
        return super.price()+30;
    }
}

这样一个完整的蛋糕就出来了，客户可以这样来购买到蛋糕：

public class Main {
    public static void main(String[] args) {
        Cake cake = new CakeWithCreamFruit(); //客户购买蛋糕
        System.out.println(cake.getDesc());
        System.out.println(cake.price());
    }
}


通过继承的方式，是可以实现装饰的定义——在不改变原有对象的基础上，将功能附加在对象上。但只能在业务简单的情况下实现，一旦业务场景变得复杂，就会出现类爆炸的情况。例如商家想要推出其他配方的蛋糕，有加巧克力的，有加芝士的，有加坚果的…每一个装饰着可以互相组合装饰到蛋糕上。这样就会出现大量的实现类，逻辑复杂，因此通过装饰者模式可以解决这种复杂的业务场景。

装饰者模式实现

首先明确装饰对象和被装饰对象。在上述场景中，被装饰对象为Cake，装饰对象为Cream和Fruit。

然后实现装饰者模式需要有四个重要的特征：

  • 抽象出被装饰者
  • 抽象出装饰者接口
  • 装饰者接口继承被装饰者的抽象类
  • 装饰者接口组合被装饰者的抽象类，并通过构造器来传入

对上述场景来逐一分析实现，首先完成被装饰者的部分，只有第一点 抽象出被装饰者 是关于被装饰者的，这个实现比较容易理解：

//抽象蛋糕
public abstract class AbstractCake {

    //蛋糕配方
    public abstract String getDesc();

    //蛋糕价格
    public abstract int price();
}
//具体的蛋糕
public class Cake extends AbstractCake {

    @Override
    public String getDesc() {
        return "蛋糕";
    }

    @Override
    public int price() {
        return 100;
    }
}

这样就在原有的基础上抽象出了蛋糕。接着实现装饰者部分，首先不难理解第二点抽象出装饰者接口，因为装饰对象可能会有很多很多，考虑到可扩展型，应该有一个统一的接口。但这里所要强调的是，此处的抽象出接口是广义上的，并非一定是一个抽象类或接口，只要突出是所有装饰者父类即可。其实现如下：

//"抽象"出装饰者
public class Decorate extends AbstractCake{

    private AbstractCake cake;  //组合

    public Decorate(AbstractCake cake) {
        this.cake = cake;
    }

    @Override
    public String getDesc() {
        return cake.getDesc();
    }

    @Override
    public int price() {
        return cake.price();
    }
}

可以看出装饰者既继承了抽象蛋糕又组合了抽象蛋糕。满足了第三点和第四点，具体为什么要这样实现，一会再说。接下来就是具体的装饰对象：

//具体奶油装饰者
public class CreamDecorate extends Decorate {

    public CreamDecorate(AbstractCake cake) {
        super(cake);
    }

    @Override
    public String getDesc() {
        return super.getDesc()+ " 加一点奶油";
    }

    @Override
    public int price() {
        return super.price()+ 20;
    }
}
//具体水果装饰者
public class FruitDecorate extends Decorate{

    public FruitDecorate(AbstractCake cake) {
        super(cake);
    }

    @Override
    public String getDesc() {
        return super.getDesc()+" 加一点水果";
    }

    @Override
    public int price() {
        return super.price()+30;
    }
}

这时，就完成了基本的装饰者模式，然后来看看顾客如何实现购买的：

public class Main {
    public static void main(String[] args) {
        AbstractCake cake = new Cake();  //获得基础蛋糕
        cake = new CreamDecorate(cake);  //加点奶油
        cake = new CreamDecorate(cake);  //再加点奶油
        cake = new FruitDecorate(cake);  //加点水果
        System.out.println(cake.getDesc());
        System.out.println(cake.price());
    }
}


可以看到，首先new一个Cake得到基础蛋糕。接着通过构造器传入这个cake，这是组合的目的出来了：组合里的蛋糕对象其实才是真正的被装饰者，通过传入新的cake可以不断装饰这个组合对象。

而继承的目的就是使装饰者是AbstractCake类型，这样可以让cake重新指向这个装饰者拿到那个最终的组合对象，getDesc()和price()实际调用的就是这个组合对象的对应方法。通过这种方式可以不断传入cake来装饰然后返回使cake重新指向，实现不断的装饰组合对象。
