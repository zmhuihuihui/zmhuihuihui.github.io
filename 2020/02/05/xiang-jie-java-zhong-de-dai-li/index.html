详解Java中的代理
代理
Java
发布日期:   2020-02-05
文章字数:   2.3k
阅读时长:   8 分

代理（Proxy）是通过代理对象访问目标对象，可以在目标对象基础上增强额外的功能，如添加权限，访问控制和审计等功能。代理分为静态代理和动态代理，动态代理又分为JDK代理和CGLib代理。

静态代理

静态代理与动态代理的区别在于，静态代理在程序运行之前，代理类的.class的文件就已经生成。何为代理类和被代理类？接下来通过一个支付宝的支付功能模拟一下。
在网上购物时，通常支付宝的支付功能，假设支付宝最简单的支付功能如下：

class AliPay{
    //支付
    public void pay(){
        System.out.println("支付中....支付成功！");
    }
}

每当用户想要购买东西时，就会使用AliPay的pay方法来支付，因此将我们要使用的目标类作为被目标类（被代理类），即Alipay。我们发现，如果只用Alipay的pay方法来完成支付，那么任何人拿到手机都可以直接支付，没有安全措施。因此，我们应该包装一下pay方法，使它更加安全，于是需要代理类来代替目标类实现相应功能。

class AliPayProxy{
    //被代理对象
    private AliPay aliPay;

    public AliPayProxy(AliPay aliPay){
        this.aliPay = aliPay;
    }

    //代理后的pay
    public void pay(){
        System.out.println("人脸识别....识别成功！");
        aliPay.pay();
        System.out.println("添加支付日志");
    }
}

通过构造器将被代理对象注入，丰富过后的pay方法同样可以实现目标类（Alipay）的支付功能，在此基础上还增加来人脸识别确保是本人支付，同时在支付成功后添加相应的日志信息。对比目标类的支付功能，增强了额外的功能，使用户直接与代理类交互，这就是静态代理模式。可以看出，静态代理中的代理类是由程序员完成的，因此在编译时代理类就会被编译，就会生成.class文件，这是静态代理的最重要的特点。

动态代理

看似静态代理可以完美的增强代理类的功能，但缺点也是十分明显的。例如在Alipay中不仅有支付功能，还有修改密码、提现等等其余功能，这些功能同样也需要增强（进行人脸识别先确认是否为本人，再记录日志）。那么，如果还使用静态代理就需要在AliPayProxy中对其余所有方法进行重新重写封装，这是十分不合理的。在这种情况下就要用到动态代理来解决。

动态代理与静态代理不同，代理类在程序运行过程中动态的生成的。动态代理分为JDK代理和Cglib代理。

JDK代理

在java.lang.reflect包下，提供了Proxy类和InvocationHandler接口，用于实现JDK代理。JDK代理中最为关键的一点是被代理类必须实现接口。
其JDK动态代理实现的步骤如下：
1、创建目标类，由于在JDK代理中目标类必须实现接口，这里做一下改变，同时增加一个修改密码的功能：

interface IAliPay{
    void pay();
    void modifyPassword();
}

class AliPay implements IAliPay{
    //支付
    public void pay(){
        System.out.println("支付中....支付成功！");
    }

    //修改密码
    public void modifyPassword(){
        System.out.println("修改密码成功!");
    }
}

2、创建代理类，代理类需要实现InvocationHandler接口,并重写其中的invoke方法。

class AliPayProxy implements InvocationHandler{
    //被代理对象
    private Object aliPay;

    public AliPayProxy(AliPay aliPay){
        this.aliPay = aliPay;
    }

    /**
     *
     * @param proxy 被代理对象
     * @param method 被调用的方法
     * @param args 方法中所需的参数
     */
    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        System.out.println("人脸识别....识别成功！");
        Object rst = method.invoke(aliPay, args);  //反射
        System.out.println("添加支付日志");
        return rst;
    }
}

3、使用Proxy类完成动态代理

public class Main {
    public static void main(String[] args) {
        InvocationHandler handler = new AliPayProxy(new AliPay());
        IAliPay aliPay = (IAliPay) Proxy.newProxyInstance(
                IAliPay.class.getClassLoader(),
                new Class[]{IAliPay.class},
                handler
        );
        aliPay.pay();
        aliPay.modifyPassword();
    }
}

结果和预期的一样，支付功能和修改密码功能同时获得了增强。可以看到，不同于静态代理，用户不是代理类（AliPayProxy）交互，而是直接和目标类（AliPay）交互，那么这一过程是如何实现的呢？

首先，创建了一个代理类的一个实例handler，并将被代理对象注入。其次，通过Proxy.newProxyInstance方法返回了一个aliPay对象。最后，直接调用目标对象的方法，我们发现最终调用的是代理类中的invoke方法。但在代码中，从头到尾没有显式的调用过invoke方法，所有的动态代理过程的实现都在Proxy.newProxyInstance中完成，接下来仔细研究一下Proxy.newProxyInstance的实现原理。

/**
* loader ： 代理类的类加载器
* interfaces ：代理类要实现的接口列表
* h ： 实现InvocationHandler接口的代理类
*/
public static Object newProxyInstance(ClassLoader loader,Class<?>[] interfaces,InvocationHandler h)
       throws IllegalArgumentException
{
       Objects.requireNonNull(h);

       final Class<?>[] intfs = interfaces.clone();
       final SecurityManager sm = System.getSecurityManager();
       if (sm != null) {
           checkProxyAccess(Reflection.getCallerClass(), loader, intfs);
       }

       //创建代理对象$proxy0
       Class<?> cl = getProxyClass0(loader, intfs);
       try {
           if (sm != null) {
               checkNewProxyPermission(Reflection.getCallerClass(), cl);
           }

           //通过反射，将代理类h通过构造器实例化代理对象$proxy0
           final Constructor<?> cons = cl.getConstructor(constructorParams);
           final InvocationHandler ih = h;
           if (!Modifier.isPublic(cl.getModifiers())) {
               AccessController.doPrivileged(new PrivilegedAction<Void>() {
                   public Void run() {
                       cons.setAccessible(true);
                       return null;
                   }
               });
           }
           //返回代理对象
           return cons.newInstance(new Object[]{h});
       } catch (IllegalAccessException|InstantiationException e) {
           throw new InternalError(e.toString(), e);
       } catch (InvocationTargetException e) {
           Throwable t = e.getCause();
           if (t instanceof RuntimeException) {
               throw (RuntimeException) t;
           } else {
               throw new InternalError(t.toString(), t);
           }
       } catch (NoSuchMethodException e) {
           throw new InternalError(e.toString(), e);
       }
}

在注释处可以看到，首先会根据传入的类加载器和接口列表创建处一个代理对象$proxy0,接着利用反射有参构造器将代理类（AliPayProxy）的实例注入到代理对象$proxy0中，最后返回代理对象$proxy0。通过反编译$proxy0的字节码文件，可以解释：
1、$proxy0实际继承了Proxy类，并实现目标接口
2、当我们调用目标类对应方法时（如pay方法），其实调用的是$proxy0想对应的方法（如pay方法），而在方法内部是调用了其父类（Proxy）中属性h的invoke方法（即super.h.invoke()），而h又是之前构造器注入的代理类，因此就可以解释是如何进行动态代理以及JDK代理是如何隐式的调用invoke方法了。

CGLib代理

上述JDK代理是基于接口的代理，使得目标类和代理类实现同一个接口；而CGLib代理不同，目标类可以是类，只需代理类继承目标类。
其CGLib动态代理实现步骤如下：
1、创建目标类

class AliPay {
    //支付
    public void pay(){
        System.out.println("支付中....支付成功！");
    }

    //修改密码
    public void modifyPassword(){
        System.out.println("修改密码成功!");
    }
}

2、实现方法拦截器，重写intercept方法(类似与实现InvocationHandler接口，但此并非代理类)

class MyMethodInterceptor implements MethodInterceptor {

    /**
     * @param o 被代理对象
     * @param method 被调用的方法
     * @param objects 方法中所需的参数
     * @param methodProxy 代理方法
     */
    @Override
    public Object intercept(Object o, Method method, Object[] objects, MethodProxy methodProxy) throws Throwable {
        System.out.println("人脸识别....识别成功！");
        Object rst = methodProxy.invokeSuper(o,objects);
        System.out.println("添加支付日志");
        return rst;
    }
}

3、使用Enhancer类完成动态代理

public class JavaSE_26 {
    public static void main(String[] args) {
        //创建Enhancer对象（相当于Proxy类）
        Enhancer enhancer = new Enhancer();
        //设置被代理对象
        enhancer.setSuperclass(AliPay.class);
        //设置回调函数，注入方法拦截器
        enhancer.setCallback(new MyMethodInterceptor());
        //生成代理对象aliPayProxy
        AliPay aliPayProxy = (AliPay) enhancer.create();
        aliPayProxy.pay();
        aliPayProxy.modifyPassword();
    }
}

在CGlib中，我们没有定义代理类而是自动从enhancer对象中create出了一个代理对象。在create方法中，官方是这样注释的：在必要时会创建一个新类，并使用指定的回调对象创建出一个实例。这里的实例也就是代理对象aliPayProxy，而当调用方法时就会自动调用注入的拦截器的intercept方法，从而实现动态代理，这和JDK代理有异曲同工之妙。

JDK代理与CGLib代理的区别

在了解完了两种动态代理的原理后，那么这两种方法有什么样的区别，在不同的场合下应该使用哪一种动态代理呢？

  • JDK代理
    1、通过利用实现InvocationHanlder接口和反射实现动态代理，目标对象必须实现接口。

  • CGLib代理
    1、通过利用实现MethodInterceptor拦截器来完成方法的拦截和代理，直接生成代理类的字节码生成实例，因此不需要目标对象实现接口。
    2、由于CGLib是基于继承实现的，所以无法对 static和final类 进行代理，无法对 static和private 方法进行代理。

在Spring中AOP思想和bean的创建都是通过动态代理来实现的，在Spring中是这样来选择JDK代理还是CGLib代理的：
如果目标对象实现了接口就使用JDK代理；如果目标对象没有实现接口就使用CGLib代理；如果目标对象实现了接口，且强制使用CGLib代理，就使用CGLib代理。
